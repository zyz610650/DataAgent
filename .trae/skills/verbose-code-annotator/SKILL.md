# Verbose Code Annotator (后端深度注释与学习教程生成器)

本技能不仅用于加注释，还用于产出一份可执行的代码学习教程。目标是让新同学能够按照明确顺序，从系统接口入口逐步读懂 `data-agent-management` 后端代码，并在源码中沉淀高质量中文注释。

## 核心定位
- 角色定位：代码讲解者 + 架构导读者 + 注释工程师。
- 主要产物：插入注释后的完整后端代码 + 按模块的学习路径说明。
- 适用对象：新入职同事、跨团队接手人员、需要快速熟悉旧系统的后端开发者。

## 何时调用
- 用户要求给后端代码加非常详细中文注释。
- 用户要求像教程一样讲解 data-agent-management，从哪里开始看。
- 用户要求梳理接口入口、模块顺序、调用链路、中间件作用。

## 输入范围识别
- 默认范围：仅 `data-agent-management` 后端代码。
- 支持范围：`data-agent-management` 全量、其下指定目录、指定后端文件列表。
- 若用户未指定范围：从 `data-agent-management` 的启动入口与路由入口开始，逐步扩展。
- 若目录过大：按后端模块分批处理，每批独立输出文件列表和学习建议。

## 学习导向执行总流程

### 第 0 步：确定学习目标与边界
- 明确本轮目标是快速上手业务还是深入核心链路。
- 明确输出粒度：只做核心模块，还是覆盖全量模块。
- 记录范围：`data-agent-management` 全仓、子目录或文件列表。

### 第 1 步：先找接口入口
- 优先识别：
  - Web 路由注册点（如 `router`, `routes`, `controller` 目录）。
  - 应用启动入口（如 `main`, `app`, `server` 初始化）。
  - 对外暴露 API（REST/GraphQL/RPC 定义）。
- 必须先回答两个问题：
  - 用户请求是从哪个接口进入系统的？
  - 该入口最终落到哪些核心服务方法？

### 第 2 步：建立后端模块地图
- 按职责分层：
  - 入口层（Router/Controller/Handler）
  - 业务层（Service/Domain）
  - 数据层（Repository/DAO/Mapper）
  - 基础设施层（MQ、缓存、搜索、网关、SDK 适配器）
  - 支撑层（配置、日志、鉴权、监控、工具库）
- 为每层标注：
  - 核心目录
  - 关键文件
  - 与其他层的调用关系

### 第 3 步：确定推荐阅读顺序（必须输出）
- 默认推荐顺序：
  1. 启动与路由入口（看系统如何接收请求）
  2. 关键 Controller/Handler（看请求如何分发）
  3. 核心 Service（看业务规则与状态流转）
  4. Repository/DAO（看数据落库与查询策略）
  5. 中间件与外部 API 适配层（看系统边界与依赖）
  6. 配置与运维支撑（看环境差异与可观测性）
- 每一步都要给出为什么先看它的解释。
- 若存在多个业务域：先主链路域（高频接口），再旁链路域（管理、统计、补偿任务）。

### 第 4 步：按学习顺序写注释
- 先注释核心链路，再补充边缘模块。
- 每个关键函数/类至少覆盖以下维度：
  - 功能概述：业务目标与系统角色。
  - 输入输出：参数含义、返回值语义、异常/空值情况。
  - 调用关系：上游调用方、下游依赖方。
  - 业务背景：对应业务实体与流程阶段。
  - 设计考虑：性能、安全、一致性、可维护性权衡。
  - 注意事项：仅在存在真实风险时标注潜在坑点（可选，不强制）。

### 第 5 步：重点处理中间件与外部 API
- 凡是出现中间件或外部 API/SDK 调用，注释必须补充：
  - 该依赖在架构中的位置与职责。
  - 关键参数与返回结构。
  - 常见异常、重试、幂等、超时、限流策略。
  - 文档引用（优先仓库内文档路径，其次公开链接）。
- 推荐引用格式：
  - `参考文档：<标题>：<URL>`
  - `内部文档：docs/<path>.md`

### 第 6 步：文件级一致性校验
- 术语统一：同一概念命名保持一致。
- 注释真实性：与当前代码行为一致，不臆测不存在逻辑。
- 噪音控制：简单 getter/setter 或直观代码不过度注释。
- 变更约束：不改业务逻辑，不做重构，仅增删注释和少量排版优化。

## 输出规范（必须遵守）
- 输出已插入注释后的完整代码，不能只给注释片段。
- 大仓库分批输出时，每批必须包含：
  - 当前处理文件列表。
  - 当前批次在 `data-agent-management` 中所属模块。
  - 本批推荐阅读顺序。
  - 本批重点补充的中间件/API 注释点。
  - 发现的疑似逻辑不一致或待确认问题。

## 每批输出模板

### A. 学习路径摘要
- 本批入口：`<接口/任务入口>`
- 推荐阅读顺序：
  1. `<文件1>`：`<先看原因>`
  2. `<文件2>`：`<再看原因>`
  3. `<文件3>`：`<最后看原因>`

### B. 注释变更摘要
- 文件：`<path>`
- 增强点：`<业务背景/调用关系/设计权衡/中间件说明>`
- 文档引用：`<内部文档路径或URL>`

### C. 风险与待确认项（按需）
- 若存在：写明代码行为与注释预期不一致的位置与原因。
- 若不存在：明确标注本批未发现明显逻辑冲突。

## 注释风格要求（后端优先）
- Java：Javadoc + 关键行内注释。
- Go：函数/类型前 `//` 注释 + 关键流程行内注释。
- Python：Docstring + 关键 `#` 注释。
- 其他后端语言：遵循该语言社区主流注释规范。
